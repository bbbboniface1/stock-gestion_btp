PARTIE A — VÉRIFICATION "RÉEL VS VITRINE" (CRITIQUE)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Pour CHAQUE fonctionnalité ci-dessous, vérifie dans le code (pas en supposant) :
1. Que la donnée affichée vient bien d'un appel HTTP réel vers l'API (pas un mock, pas un tableau hardcodé, pas un fallback silencieux vers des données statiques)
2. Que l'action déclenche bien une écriture en base (INSERT/UPDATE réel, pas une mise à jour optimiste qui ne persiste jamais)
3. Que rafraîchir la page (F5) après l'action montre bien la donnée persistée (preuve que ce n'est pas juste du state local React)

Fonctionnalités à vérifier une par une :
- Connexion/déconnexion → token réellement émis par le backend et vérifié à chaque requête
- Création d'un produit → apparaît en base après F5
- Mouvement IN → quantity_in_stock incrémenté EN BASE (vérifie avec une requête SQL directe, pas juste via l'UI)
- Mouvement OUT → quantity_in_stock décrémenté EN BASE + refus si stock insuffisant (teste le cas limite réellement, ne te contente pas de lire le code)
- Création de projet + ajout de matériaux → liens réels en base (project_materials)
- Dashboard (KPIs, graphiques) → recalcul réel à partir des données en base, pas de valeurs figées/cache statique trop long
- Génération de facture PDF → données réelles du produit/projet au moment de la génération, pas un template avec données factices

Pour chaque point, donne le fichier exact qui prouve que c'est réel (route API + requête SQL associée), et le résultat du test manuel effectué (avant/après F5).

🚩 Signale explicitement tout endroit où tu trouves :
- Des données mockées/hardcodées encore présentes dans un composant de production
- Un fallback qui affiche une donnée par défaut en cas d'échec réseau SANS prévenir l'utilisateur (ça simule un succès qui n'en est pas un)
- Du cache trop agressif qui pourrait masquer le fait que l'écriture en base a échoué
